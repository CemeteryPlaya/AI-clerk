# Дизайн: План 2 — Профиль (шифр. ПДн из PDF) + Геолокация (офлайн-аэропорты)

- **Дата:** 2026-06-15
- **Статус:** Согласован (ожидает финальной вычитки пользователем)
- **Владелец:** Админ проекта
- **Контекст:** Продолжение проекта «Личный AI-секретарь (Telegram-бот)», модуль «Командировки».
  План 1 (фундамент: роли, онбординг, БД, крипто) завершён. Базовая спецификация:
  `docs/superpowers/specs/2026-06-15-ai-secretary-business-trips-design.md` (§3.7 ProfileService,
  §3.9 / §5 LocationService).

## 1. Назначение и объём

План 2 реализует две способности-сервиса и тонкую обвязку бота для их использования:

1. **ProfileService** — хранит профиль путешественника: зашифрованные **личные данные (ПДн)**,
   извлечённые **локально** из загруженного PDF; а также нечувствительные предпочтения, программы
   лояльности и политику/лимиты.
2. **LocationService** — встроенный (офлайн) справочник аэропортов, поиск **ближайшего аэропорта** по
   координатам, поиск аэропорта по городу/IATA и резолвер точки отправления по цепочке приоритетов
   (спека §5).

### В объёме Плана 2
- Модель `Profile` (1:1 с `User`) с шифрованием ПДн через существующий `Cipher` (Fernet).
- Пайплайн **PDF → текст (text-layer, иначе OCR) → извлечение полей → подтверждение (маскировано) →
  шифрование и сохранение**, целиком **локально**.
- OCR за интерфейсом: реальный движок (Tesseract) только в проде/Docker; в тестах — фейк.
- Извлечение полей за интерфейсом: дефолт — детерминированный regex-экстрактор (тестируемый);
  локальный LLM — будущая замена.
- `LocationService`: загрузка справочника аэропортов (OurAirports CSV), `nearest()` (haversine),
  `by_city()`, `by_iata()`, `resolve_departure()`; UX «поделиться геопозицией» в боте.
- Команды бота для профиля и точки отправления, под контролем доступа (`profile.edit`).

### Вне объёма Плана 2 (последующие планы / фазы)
- Применение политики/лимитов при ранжировании — План 3 (значения политики по умолчанию в спеке пока
  не заданы; здесь поля только **хранятся**).
- NLU-разбор свободного текста («лечу из Алматы») — План 3 (Claude slot-filling).
- Live-геокодер **Nominatim** (reverse-geocoding координат в город) — не нужен: ближайший аэропорт
  считается прямо по координатам.
- Локальный LLM-экстрактор полей; OCR для не-казахстанских/международных документов.
- Международные адреса, доставка по email/WhatsApp — Фаза 2.

## 2. Решения, зафиксированные на брейншторме

| Вопрос | Решение | Причина |
|---|---|---|
| Объём Плана 2 | Profile + Location вместе; аэропорты — **скачанный статический датасет** | Данные аэропортов меняются редко; нет зависимости от живого сервиса; офлайн-тестируемость |
| Ввод ПДн | Директор присылает **PDF-документ**, бот извлекает данные **локально** | Данные не вводятся текстом в чат; обработка локальная; этот класс данных и так широко доступен |
| Извлечение из PDF | **OCR сейчас** (скан/фото-PDF), но движок OCR — за интерфейсом | Поддержать реальные документы; фейк OCR в тестах, чтобы не требовать Tesseract на dev-машине (Windows) |
| Модель путешественника | **1:1** `Profile` ↔ `User`; путешественник = владелец профиля | MVP для директора; бронь за других — позже |
| Шифрование | ПДн (ФИО, ИИН, № документа, дата рожд.) — шифр.; предпочтения/политика — открытым текстом | Спека §6: ПДн at-rest зашифрованы; нечувствительное — для поиска/отображения |
| Политика/лимиты | В Плане 2 только **хранятся** (nullable), не применяются | Значения политики ещё не заданы; применение — План 3 |
| Подтверждение ПДн | Бот показывает **маскированные** значения + кнопки «Сохранить / Загрузить заново» | Не пере-засвечивать полные ПДн в чат |
| Геокодер | **Без Nominatim** в Плане 2 | Ближайший аэропорт считается по координатам без названия города |

## 3. Модель данных (`db/models.py`)

Новая таблица `profiles`, связь 1:1 с `users` (`user_id` уникальный FK на `users.id`). Стиль —
как в существующих моделях: `Mapped`/`mapped_column`, timestamps управляются Python
(`default=lambda: datetime.now(timezone.utc)`, `onupdate=...`).

**Зашифрованные ПДн** (хранятся как Fernet-токены, тип `Text`/`String`; шифрует/дешифрует
`ProfileService` через `Cipher`):
- `full_name_enc` — ФИО
- `iin_enc` — ИИН (12 цифр)
- `document_number_enc` — номер паспорта/УДО
- `birth_date_enc` — дата рождения (хранится как зашифрованная строка ISO)

**Открытым текстом, нечувствительное:**
- `document_type` — `passport` | `udo`
- `position` — должность (нужна для приказа)
- `citizenship` — гражданство (по умолчанию KZ)

**Предпочтения (открытым текстом):**
- `default_departure_iata` (`String(8)`, nullable) и `default_departure_city` (nullable)
- `preferred_airlines` — JSON-список
- `preferred_hotels` — JSON-список
- `seat_preference` — например, окно/проход
- `meal_preference`
- `prefer_faster` — `bool`, по умолчанию `True` («время в пути приоритетнее»)

**Лояльность (открытым текстом, JSON):** список `{program, number}`.

**Политика/лимиты (открытым текстом, все nullable):**
- `budget_limit` — лимит бюджета на поездку
- `cabin_class` — класс обслуживания
- `hotel_max_stars` — макс. звёздность
- `per_diem` — суточные

`created_at` / `updated_at` — как в `models.py`.

### Контракт ProfileService
`ProfileService(session, cipher)`:
- `get_profile(user_id) -> ProfileDTO | None` — возвращает **дешифрованный** DTO (dataclass), чтобы
  детали шифрования не выходили за границу сервиса.
- `upsert_identity(user_id, full_name, iin, document_type, document_number, birth_date, position=None,
  citizenship=None) -> ProfileDTO` — шифрует ПДн и сохраняет (upsert).
- `set_preferences(user_id, **prefs)` / `set_policy(user_id, **limits)` /
  `set_default_departure(user_id, iata=None, city=None)`.

В БД лежит шифртекст; чтение возвращает дешифрованные значения. Маскировка для отображения — отдельная
утилита (`mask_iin`, `mask_document`), не хранит и не логирует полные значения.

## 4. Пайплайн «PDF → зашифрованный профиль» (всё локально)

Слои за интерфейсами: пайплайн офлайн-тестируем, тяжёлые нативные зависимости живут только в проде.

```
PDF (bytes) ─▶ PdfTextExtractor ─▶ ProfileExtractor ─▶ ExtractedProfile
                (сначала text-layer;   (RegexProfileExtractor   (маскир. подтверждение
                 иначе OCR-fallback)     по умолчанию;            ▶ шифрование + запись)
                                         LLM — позже)
```

- **`PdfTextExtractor(ocr_engine)`** — сначала пытается извлечь текстовый слой PDF; если текста нет
  или он слишком короткий → OCR-ветка через `ocr_engine`.
- **`OcrEngine`** (интерфейс) — `recognize_pdf(pdf_bytes, langs=("rus","kaz","eng")) -> str`.
  - `TesseractOcrEngine` — прод-адаптер: растеризация страниц + Tesseract (`rus`+`kaz`+`eng`).
  - `FakeOcrEngine(text=...)` — для юнит-тестов; Tesseract в тестах не требуется.
- **`ProfileExtractor`** (интерфейс) — `extract(text) -> ExtractedProfile`.
  - `RegexProfileExtractor` (дефолт) — детерминированный: ИИН = 12 цифр, паттерны номера документа и
    дат, эвристики для ФИО. Полностью тестируем.
  - Локальный LLM-экстрактор — будущая замена за тем же интерфейсом.
- **`ExtractedProfile`** — dataclass с возможно-отсутствующими полями (что не распозналось → `None`).
- **Подтверждение:** хэндлер бота отвечает **маскированными** извлечёнными значениями
  (`ИИН ••••••••1234`, `№ док. •••567`) и inline-кнопками **«Сохранить» / «Загрузить заново»**.
  По «Сохранить» → `ProfileService.upsert_identity(...)` (шифрование + запись). Полные ПДн в чат не
  выводятся.

Скачивание файла из Telegram и весь разбор — локально на сервере; в облако (Claude) ничего из этого
не уходит (спека §6).

## 5. LocationService (офлайн, без живого геокодера)

- **`AirportIndex`** — загружает встроенный **OurAirports `airports.csv`** (public domain), для MVP
  отфильтрованный по Казахстану (загрузчик умеет принять и полный набор позже). Методы:
  - `nearest(lat, lon) -> Airport` — ближайший по формуле **haversine**.
  - `by_city(name) -> Airport | None` — нормализация + **таблица алиасов KZ**
    (`Алматы/Almaty`, `Астана/Astana/Nur-Sultan`, `Шымкент/Shymkent`, и т.д.) + сопоставление с
    `municipality`.
  - `by_iata(code) -> Airport | None`.
  - `Airport` — dataclass: `iata`, `name`, `city`, `lat`, `lon`, `country`.
- **`resolve_departure(explicit_city, coords, profile_default) -> DepartureResolution | None`** —
  чистая функция, реализует цепочку §5:
  1. явный город (если передан) → `by_city`;
  2. координаты (геопозиция) → `nearest`;
  3. город вылета по умолчанию из профиля → `by_city`;
  4. иначе → `None` (сигнал «нужно уточнить / попросить геопозицию»).
  Возвращает выбранный `Airport` и `source` (откуда взято). Разбор «лечу из Алматы» из свободного
  текста — задача оркестратора (План 3); сюда `explicit_city` приходит уже как параметр.
- **UX в боте:** кнопка Telegram **`request_location`** → ближайший аэропорт → предложение
  **«запомнить как город вылета по умолчанию»**. Координаты **не сохраняются**, если директор явно не
  согласился (спека §5).

## 6. Команды / UX бота (тонкая обвязка, доступ `profile.edit` — DIRECTOR/ADMIN)

- `/profile` — сводка: **маскированные** ПДн + предпочтения + политика; точка входа в редактирование.
- «Заполнить документы» → просьба прислать PDF → пайплайн из §4.
- Лёгкие команды для нечувствительных полей (город вылета по умолчанию, предпочтения, политика) +
  кнопка геопозиции из §5.
- Контроль доступа: действие `profile.edit` уже есть в `bot/permissions.py` для DIRECTOR/ADMIN;
  ACCOUNTANT профиль не редактирует. Директор редактирует **свой** профиль.

Хэндлеры остаются тонкими (вся ветвящаяся логика — в сервисах), проверка — ручным прогоном бота, как
в Плане 1 (Задачи 10–11).

## 7. Стратегия тестирования (TDD, всё офлайн)

Юнит-тесты:
- **ProfileService:** шифртекст в БД ≠ открытый текст; роундтрип дешифрования; upsert/обновление;
  предпочтения и политика.
- **RegexProfileExtractor:** извлечение ИИН/номера/ФИО/дат из примеров текста; отсутствующие поля → `None`.
- **PdfTextExtractor:** извлечение из text-layer (фикстура-PDF) + ветка OCR-fallback через `FakeOcrEngine`.
- **AirportIndex:** корректность haversine-`nearest`; `by_city` (+ алиасы) и `by_iata` на фикстуре-CSV.
- **resolve_departure:** каждая ветка цепочки приоритетов + фоллбэк «уточнить».
- **Маскировка:** `mask_iin`/`mask_document`.

Интеграционно:
- **`TesseractOcrEngine`** — тест помечен skip-if-unavailable (исполняется при наличии Tesseract);
  реальный OCR проверяется в Docker.

Хэндлеры бота — тонкие; логика покрыта тестами сервисов.

## 8. Новые зависимости и развёртывание

- **PDF/OCR:** библиотека для PDF (текст + растеризация) и `pytesseract`. Точный выбор библиотеки PDF
  (PyMuPDF vs `pypdf` + `pdf2image`) фиксируется в плане реализации; PyMuPDF покрывает и текст, и
  растеризацию одной зависимостью (лицензия AGPL — приемлемо для приватного внутреннего инструмента).
- **Docker-образ** добавляет системные пакеты: `tesseract-ocr`, `tesseract-ocr-rus`,
  `tesseract-ocr-kaz`, `tesseract-ocr-eng` (+ `poppler-utils`, если выбран `pdf2image`).
- **Данные:** встроенный `airports.csv` (OurAirports, public domain), отфильтрованный по KZ, в репозитории.
- **Схема БД:** новая таблица создаётся существующим `Base.metadata.create_all` (Alembic пока не
  вводим — приемлемо для MVP; миграции — отдельный вопрос на будущее).

## 9. Покрытие спецификации

- ProfileService (спека §3.7): шифр. ПДн, предпочтения, лояльность, политика/лимиты, приоритет времени
  в пути (`prefer_faster`) — §3 этого документа. ✓
- LocationService (спека §3.9, §5): ближайший аэропорт, цепочка приоритетов, разовая геопозиция без
  постоянного трекинга — §5. ✓ (Nominatim сознательно отложен; ближайший аэропорт по координатам.)
- Безопасность (спека §6): ПДн шифруются at-rest; обработка PDF локальная, в облако не уходит;
  маскировка при отображении. ✓
- Контроль доступа (спека §3.8): `profile.edit` для DIRECTOR/ADMIN. ✓
- Отложено осознанно (в последующие планы): применение политики при ранжировании (План 3),
  NLU-разбор города (План 3), live-Nominatim, LLM-экстрактор, международные документы/адреса.

## 10. Открытые вопросы (не блокируют реализацию)

- Точный набор полей казахстанских документов и их типовые надписи в PDF — уточняется по реальным
  образцам; regex-экстрактор стартует с ИИН/ФИО/№ документа/даты рождения.
- Выбор PDF-библиотеки (PyMuPDF vs pypdf+pdf2image) — финализируется в плане.
- Источник/версия датасета аэропортов и периодичность обновления (датасет редко меняется).

# Дизайн: План 3 — Оркестрация поездки + поиск

- **Дата:** 2026-06-16
- **Статус:** Согласован (ожидает финальной вычитки пользователем)
- **Владелец:** Админ проекта
- **Контекст:** Продолжение проекта «Личный AI-секретарь (Telegram-бот)». Планы 1 (фундамент) и 2
  (профиль + геолокация) завершены и в `main`. Базовая спецификация:
  `docs/superpowers/specs/2026-06-15-ai-secretary-business-trips-design.md` (§2 решения, §3 компоненты
  Dialog Orchestrator / Trip Saga / BookingProvider, §4 ранжирование, §6 безопасность).

## 1. Назначение и объём

План 3 реализует разговорную часть командировки: от свободного запроса директора до **подтверждённого
выбора варианта**.

### В объёме Плана 3
Цепочка: **сбор требований (Claude slot-filling) → определение точки отправления (существующий
LocationService) → поиск (BookingProvider) → ранжирование → показ топ-вариантов → подтверждение в один
тап → сохранение `Trip`**.

### Вне объёма Плана 3 (последующие планы)
- Реальная покупка/бронь, браузер-агент, `book()`, OTP — **План 4**.
- Полностью персистентная сага (restart-safety каждого шага) — **План 4** (здесь диалог в памяти).
- Реальные провайдеры (Aviata/Kaspi) — **План 4**; здесь только `MockProvider`.
- Учёт времени доезда до аэропорта в ранжировании — позже (нет данных о транспорте).
- Генерация приказа — **План 5** (ПДн тянутся из `ProfileService` в момент генерации).

**Граница:** План 3 завершается записью `Trip(status=CONFIRMED)` — это устойчивая точка передачи
управления саге бронирования Плана 4.

## 2. Решения, зафиксированные на брейншторме

| Вопрос | Решение | Причина |
|---|---|---|
| «Мозг» оркестратора | Интерфейс `LlmClient` + `FakeLlmClient` для тестов + реальный `ClaudeClient` (Anthropic SDK) для прода | Детерминированная стейт-машина тестируется офлайн; ключ API не нужен для прогона тестов; паттерн «провайдер за интерфейсом + фейк» как в коде |
| Роль Claude | **Только NLU + диалог** (понимание запроса, уточняющие вопросы, заполнение слотов); оркестрацию инструментов делает Python | Чище и тестируемее, чем Claude-driven tool-calling; tool-calling — будущая опция |
| Персистентность саги | Диалог **в памяти** per-chat; `Trip` пишется в БД **в момент подтверждения** | YAGNI; restart-safety критичен для оплаты (План 4), не для сбора требований; даёт Плану 4 устойчивую точку старта |
| Ранжирование | По **длительности перелёта** (вылет→прилёт с пересадками), затем по **цене в рамках политики** | «Время-первично» из спеки; доезд требует транспортных данных, которых нет; ближайший аэропорт уже выбран LocationService |
| Применение политики | Фильтр/штраф по бюджету и классу **только если поля политики заданы** в профиле | Значения политики по умолчанию ещё не заданы (спека) |
| Вход в диалог | Любой **не-командный текст** от пользователя с правом `trip.create` (DIRECTOR/ADMIN) → оркестратор; `/cancel` сбрасывает диалог | Секретарь, с которым «просто разговаривают»; отдельная команда `/trip` не нужна |
| Кол-во вариантов | Топ-**3**, inline-кнопки, подтверждение в один тап | Достаточно для выбора, не перегружает чат |
| ПДн и облако | В Claude уходит **только трип-контекст** (направление/даты/предпочтения), ИИН/паспорт/ФИО — **никогда** | §6 спеки выполняется «по построению» |

## 3. Компоненты (новый пакет `trips/` + `bot/trip_handlers.py`)

1. **`LlmClient` (интерфейс, `trips/llm.py`)** — `fill_slots(history, known) -> SlotFillResult`
   (обновлённый `TripRequest`, список недостающих полей, ответ ассистента/уточняющий вопрос).
   - `FakeLlmClient` — скриптуемый, для юнит-тестов (детерминированные результаты).
   - `ClaudeClient` — прод-адаптер на Anthropic SDK; модель и ключ из настроек. Тонкий слой,
     проверяется вручную (как OCR в Docker), юнит-тестами не покрывается.
2. **`Orchestrator` (`trips/orchestrator.py`)** — детерминированная диалоговая стейт-машина поверх
   in-memory состояния per-chat. На каждое сообщение: вызывает `LlmClient.fill_slots`, накапливает
   `TripRequest`; пока есть недостающие слоты — отдаёт уточняющий вопрос; когда слоты заполнены —
   запускает поиск→ранжирование→показ. Полностью покрыт юнит-тестами с `FakeLlmClient`.
3. **`BookingProvider` (интерфейс, `trips/provider.py`)** — `search_flights(origin_iata, dest_iata,
   date, ...)`, `search_hotels(city, checkin, checkout, ...)`, `book(...)`. Датаклассы `FlightOption`
   (id, авиакомпании/сегменты, время вылета/прилёта, общая длительность, цена, класс) и `HotelOption`
   (id, название, звёздность, цена/ночь, итог, адрес).
   - `MockProvider` (`trips/mock_provider.py`) — детерминированные варианты по KZ для тестов/дева;
     `book()` бросает `NotImplementedError` (План 4).
4. **`ranking.py`** — `rank_flights(options, policy)`: сортировка по длительности, затем по цене;
   фильтр/штраф вариантов сверх политики (бюджет/класс) **только если поля политики заданы**.
5. **`TripService` (`trips/service.py`)** — сохраняет подтверждённый `Trip` в БД.
6. **`bot/trip_handlers.py`** — роутер: не-командный текст → `Orchestrator`; показ вариантов
   inline-кнопками; callback подтверждения → `TripService`. Подключается в `main.py`, сервисы —
   через middleware/синглтоны.

Каждый компонент имеет одну ответственность и общается через явный интерфейс; тяжёлая/недетерминированная
часть (Claude) изолирована за `LlmClient`.

## 4. Данные

- **Модель `Trip`** (новая таблица): `id`, `chat_id`, `telegram_user_id`, `origin_iata`,
  `dest_city`, `dest_iata`, `depart_date`, `return_date` (nullable = one-way), `selected_flight`
  (JSON), `selected_hotel` (JSON, nullable), `status` (`CONFIRMED` сейчас; новые статусы — План 4),
  `created_at`/`updated_at` (Python-managed, как в остальных моделях). **Сырые ПДн на поездке не
  хранятся** — личность остаётся в `Profile`; приказ (План 5) берёт ПДн из `ProfileService` при
  генерации.
- **`TripRequest`** (in-memory датакласс): `dest_city`, `dest_iata`, `depart_date`, `return_date`,
  `one_way`, `notes`. Точка отправления определяется через `LocationService`; предпочтения
  (авиакомпании, `prefer_faster`) берутся из `ProfileService`.
- **`SlotFillResult`** (in-memory датакласс): обновлённый `TripRequest`, `missing_fields: list[str]`,
  `assistant_message: str | None`.

## 5. Поток данных

Директор пишет свободный текст → `trip_handlers` направляет в `Orchestrator` → `LlmClient.fill_slots`
→ если есть недостающие слоты, бот задаёт уточняющий вопрос; если заполнены — `Orchestrator` вызывает
`LocationService` (отправление) + `BookingProvider.search_*`, ранжирует в рамках политики и отправляет
**топ-3** варианта сообщением с **inline-кнопками (один тап)**. Тап по варианту → `TripService`
пишет `Trip(status=CONFIRMED)` → бот подтверждает «вариант выбран; бронирование далее» (шов к Плану 4).

`/cancel` очищает in-memory диалог по чату. Перезапуск бота во время диалога = диалог начинается заново
(приемлемо для Плана 3).

## 6. Безопасность и приватность (спека §6)

- В облачный Claude уходит **только трип-контекст** (направление, даты, предпочтения). **ИИН, паспорт,
  ФИО никогда не попадают в промпт** оркестратора — маскировка PII выполняется «по построению».
- Доступ к созданию поездки — право `trip.create` (DIRECTOR/ADMIN); ACCOUNTANT не инициирует поездки.
- Ключ Anthropic — из переменных окружения; в логи не пишется.

## 7. Обработка ошибок

- Недоступность Claude (`ClaudeClient` бросил) → бот отвечает понятным сообщением «не смог обработать
  запрос, попробуйте ещё раз», диалоговое состояние сохраняется в памяти.
- Поиск не дал вариантов → бот сообщает и предлагает изменить даты/направление.
- Незаполненные слоты → оркестратор задаёт конкретный уточняющий вопрос (из `SlotFillResult`).
- `book()` в Плане 3 не вызывается; `MockProvider.book()` бросает `NotImplementedError` (страховка).

## 8. Стратегия тестирования (TDD, офлайн)

- **`Orchestrator`:** переходы «вопрос → дозаполнение → готово»; запуск поиска при полном наборе
  слотов; обработка пустого результата — со скриптуемым `FakeLlmClient`.
- **`MockProvider`:** структура и детерминизм результатов.
- **`rank_flights`:** длительность→цена; фильтрация по политике; политика не задана = без фильтра.
- **`TripService`:** запись `Trip(status=CONFIRMED)` и чтение.
- **Формирование сообщения с вариантами** (формат, кнопки) — юнит-тест на чистой функции.
- **`ClaudeClient`** — тонкий адаптер, проверяется вручную (нужен `ANTHROPIC_API_KEY`); при реализации
  свериться со справочником claude-api по актуальному id модели и использованию SDK.

## 9. Новые зависимости и настройки

- **Dep:** `anthropic` (SDK; тестовый прогон его не вызывает — только прод-адаптер).
- **Настройки (`config.py`):** `anthropic_api_key: str | None`, `anthropic_model: str` (конфигурируемо,
  по умолчанию актуальная модель Claude).
- Схема БД: новая таблица `trips` создаётся существующим `Base.metadata.create_all`.

## 10. Открытые вопросы (не блокируют реализацию)

- Точные поля `FlightOption`/`HotelOption` под будущие реальные провайдеры (Aviata/Kaspi) —
  уточнятся в Плане 4; в Плане 3 берём практичный минимум для ранжирования и показа.
- Формулировки системного промпта Claude для slot-filling — донастроятся при ручной проверке адаптера.
- Значения политики по умолчанию (бюджет/класс/звёздность) — пока не заданы (общий открытый вопрос).

# Дизайн: Личный AI-секретарь (Telegram-бот) — модуль «Командировки»

- **Дата:** 2026-06-15
- **Статус:** Согласован (ожидает финальной вычитки пользователем)
- **Владелец:** Админ проекта (ina.sanzhar@gmail.com)

## 1. Назначение и объём

Личный AI-секретарь генерального директора в виде Telegram-бота. Первая способность —
**организация командировок**. Архитектура изначально проектируется как «секретарь с набором
способностей», чтобы в дальнейшем добавлять календарь, встречи, расходы и т.д. без переписывания
ядра.

### Задачи MVP (модуль «Командировки»)
1. Понимать свободные пожелания директора (куда, когда, предпочтения) на естественном языке.
2. Определять точку отправления (геолокация по запросу / профиль / явное указание).
3. Находить билеты (самолёт предпочтительнее — «быстрее = лучше») и отели на выбранные даты.
4. Ранжировать варианты по **общему времени в пути**, затем по цене в рамках политики компании.
5. Показывать топ-варианты; директор подтверждает один в **один тап**.
6. После подтверждения бот **сам завершает покупку/бронь** через браузер-агента под аккаунтом OTA.
7. Запрашивать у директора **3DS/OTP** на этапе оплаты.
8. Генерировать **командировочный приказ** по шаблону и отправлять бухгалтеру.
9. Сохранять историю поездок в **папки-контейнеры** с хэш-именами и быстрым индексом в БД.

### Вне объёма MVP (последующие фазы)
- Доставка приказа по email/WhatsApp (Фаза 2).
- Международные направления с уточнением полного адреса (Фаза 2).
- Напоминания (регистрация, вылет), интеграции GDS/агентств (Фаза 2).
- Прочие способности секретаря: календарь, встречи, расходы (Фаза 3).

## 2. Архитектурные решения (зафиксированы)

| Решение | Выбор | Причина |
|---|---|---|
| Стиль архитектуры | Модульный монолит с подключаемыми провайдерами | Достаточно для штата до ~10 чел.; микросервисы оправданы только при высоком RPS |
| Стек | Python 3.14.x + aiogram 3.x | Скорость интерпретатора; зрелая экосистема Telegram-ботов |
| LLM | Гибрид: Claude (облако) для диалога/планирования + локальный LLM для браузер-агента | Качество понимания + приватность чувствительных операций |
| Механизм покупки | LLM браузер-агент (computer-use) под аккаунтом OTA | В КЗ нет простых публичных API для реальной оплаты билетов |
| Платёжные данные | Карта привязана в аккаунте OTA — бот не касается карты | Безопасность, минимизация PCI-ответственности |
| 3DS/OTP | Бот запрашивает код у директора в чате | Банки КЗ почти всегда требуют OTP |
| География (MVP) | Казахстан; далее международные | Поэтапный рост |
| Хостинг | VPS/сервер в Казахстане (Linux + headless-браузер) | Резидентность данных (закон РК о ПДн) |
| Хранение истории | Папки-контейнеры по чатам с хэш-именами + индекс в PostgreSQL | Быстрый поиск, доступ за O(1) по хэшу/пути |
| Роли | Хардкод: ADMIN, DIRECTOR, ACCOUNTANT; онбординг по ссылке-приглашению | Простота и контроль на старте |

### Каркас (для будущих способностей)
Ядро бота — «секретарь со способностями» (capabilities). «Командировки» — первая способность.
Каждая способность регистрирует свои команды/намерения у Orchestrator и свои сервисы. Добавление
способности не меняет Gateway, RoleService, ProfileService, архив и индекс.

## 3. Компоненты

1. **Telegram Gateway (aiogram)** — приём апдейтов, маршрутизация, проверка ролей, онбординг по
   ссылке, inline-кнопки подтверждения, запрос геолокации и OTP.
2. **Dialog Orchestrator (Claude, облако)** — NLU свободных пожеланий, заполнение слотов
   (куда/когда/предпочтения), уточняющие вопросы, tool-calling к внутренним сервисам, сборка плана.
   Хранит состояние диалога на чат. Перед отправкой в облако **маскирует PII** (см. §6).
3. **Trip Saga (сохраняемая стейт-машина)** — ведёт поездку по шагам: сбор требований → определение
   точки отправления → поиск → показ вариантов → подтверждение → бронь (билет+отель) → OTP → приказ →
   уведомление бухгалтера → архив. Переживает перезапуск; шаги идемпотентны; поддерживает паузу
   «на человека».
4. **BookingProvider (интерфейс)** — `search_flights()`, `search_hotels()`, `book()`.
   Реализации: `BrowserAgentProvider` (computer-use агент в браузере под аккаунтом OTA); на будущее —
   `ApiProvider` (GDS/агентство). **Предпочтительные сервисы — банков Freedom (Aviata/Freedom Travel)
   и Kaspi (Kaspi Travel)**, где привязана карта; покупка выполняется браузер-агентом. Лёгкие API
   (например, Travelpayouts/Aviasales) допустимы для быстрого поиска вариантов там, где доступно.
5. **Browser-Agent Worker (отдельный процесс)** — локальный LLM + Playwright. Получает задачи через
   внутреннюю очередь, отчитывается о прогрессе и событии «нужен OTP». Изолирован: краш/зависание
   браузера не роняет бот. Сессии/cookies OTA хранятся зашифрованно.
6. **OrderService** — генерирует командировочный приказ по **шаблону DOCX с размеченными местами для
   подстановки** (docxtpl/Jinja) из данных профиля и поездки; при необходимости конвертирует в PDF;
   доставляет бухгалтеру по настраиваемому каналу (Telegram сейчас; email/WhatsApp — Фаза 2).
7. **ProfileService** — хранит на путешественника: личные данные (ИИН, паспорт/УДО, ФИО, должность)
   **в шифровании**; предпочтения (город вылета, авиакомпании, отели, место/питание); программы
   лояльности; политику/лимиты (бюджет, класс, звёздность, суточные); приоритет времени в пути.
8. **RoleService / AccessControl** — хардкод-роли; онбординг по одноразовой ссылке-приглашению
   (непубличный случайный токен, Telegram deep-link-safe `[A-Za-z0-9_-]` ≤64 симв., в БД
   `pending_invites`, одноразовый + TTL); проверка прав на каждое действие.
9. **LocationService** — разовый запрос геолокации (нативная кнопка Telegram `request_location`),
   reverse-geocoding в город + ближайший аэропорт/станцию, определение точки отправления по цепочке
   приоритетов (см. §5). Без постоянного трекинга.
10. **TripArchive (контейнеры)** — папки по чатам; каждая поездка = папка-контейнер с хэш-именем; все
    артефакты + `manifest.json`.
11. **Index/DB (PostgreSQL)** — быстрый индекс поездок, состояние Saga, профили (с шифр-полями),
    роли, аудит-лог.
12. **Crypto/Secrets** — шифрование PII и сессий OTA (Fernet/libsodium), ключ из секрет-хранилища;
    аудит чувствительных действий.

## 4. Хранилище-контейнеры и индекс

```
storage/
  chats/
    <chat_hash>/                  # папка на чат (хэш идентичности чата)
      trips/
        <trip_hash>/              # контейнер; hash = sha256 канонического ключа поездки
          manifest.json           # быстрые метаданные
          order/prikaz.pdf
          flights/eticket_*.pdf
          hotel/confirmation_*.pdf
          receipts/*.pdf
          agent_log/*.json        # трасса браузер-агента (аудит/отладка)
```

- `manifest.json`: кто, куда, даты, стоимость, статусы, список файлов, `created_at`, `hash`.
- БД-индекс дублирует ключевые поля для поиска; **хэш и путь контейнера хранятся в БД** для доступа
  за O(1).
- `<trip_hash>` = sha256 от канонического ключа поездки (traveler + destination + dates + seq),
  чтобы имя было детерминированным и коллизие-устойчивым.

## 5. Определение точки отправления (LocationService)

Цепочка приоритетов:
1. Явно указано в сообщении («лечу из Алматы»).
2. Геопозиция, которой пользователь поделился по запросу (`request_location`).
3. Город вылета по умолчанию из профиля.
4. Иначе — бот уточняет и предлагает поделиться локацией.

- Координаты → город → ближайший **аэропорт** (и ж/д станция) через геокодер (**Nominatim**,
  self-host) + справочник аэропортов/станций.
- Точка отправления участвует в расчёте **общего времени в пути** (доезд до узла + перелёт/поездка),
  что и определяет ранжирование «время → цена».
- Координаты используются для активной поездки и не сохраняются постоянно, если пользователь явно не
  выбрал «запомнить как город вылета по умолчанию».

## 6. Безопасность и комплаенс

- **Карта.** Сырые данные карты не хранятся и не вводятся в боте; карта живёт в аккаунте OTA. Бот
  инициирует оплату через браузер-агента, 3DS/OTP запрашивает у директора в чате.
- **PII в шифровании.** ИИН, паспорт/УДО, ФИО — шифрование at-rest (Fernet/libsodium), ключ из
  переменных окружения/секрет-хранилища; доступ по ролям; аудит-лог.
- **Резидентность (закон РК о ПДн).** Хранилище и БД — на VPS в Казахстане. В облачный Claude уходит
  только необходимое для диалога; **PII (ИИН/паспорт) маскируется/редактируется** перед отправкой;
  чувствительные операции — локально.
- **Сессии OTA** хранятся зашифрованно; принцип наименьших привилегий.
- **Финансовый контроль.** Любая оплата — только после явного подтверждения; трата проверяется по
  лимитам политики; превышение бюджета → стоп, пока директор явно не разрешит.
- **Аудит.** Каждое внешнее действие (поиск, бронь, оплата, отправка приказа) логируется
  (кто/что/когда).

## 7. Обработка ошибок и устойчивость

- **Сага persistent.** Каждый шаг сохраняется в БД; после перезапуска поездка продолжается с того же
  места.
- **Идемпотентность.** Перед оплатой — проверка «не куплено ли уже»; повтор шага не создаёт вторую
  бронь.
- **Сбои браузер-агента** (таймаут, CAPTCHA, неуспешный логин) → эскалация человеку: сообщение +
  выбор «повтор» либо ручная ссылка для завершения.
- **OTP-таймаут / отказ оплаты** → безопасный откат/отмена, понятный статус поездки, уведомление.
- **Деградация поиска.** Если API поиска недоступен — фоллбэк на браузер-агента или варианты вручную.

## 8. Стратегия тестирования (TDD)

- **Юнит-тесты** на каждый сервис: ProfileService, OrderService, RoleService, TripArchive/индекс,
  LocationService, переходы Saga.
- **BookingProvider за интерфейсом** → мок-провайдер для тестов; браузер-агент — на записанных
  фикстурах/в staging-режиме без реальных платежей.
- **Интеграционный тест полной саги** на мок-провайдере, включая паузу на OTP и возобновление после
  перезапуска.
- **Рендер приказа** — golden-file тесты шаблона.
- **Политика/лимиты** — тесты на превышение бюджета, класс, звёздность.

## 9. Технологический стек

- **Python 3.14.x**, **aiogram 3.x**
- **PostgreSQL** (индекс, состояние саги, профили с шифр-полями); для самой первой итерации допустим
  SQLite
- **Playwright** + локальный LLM (предположительно **OpenClaw**) — браузер-агент
- **Anthropic Claude API** (claude-opus-4-8 / актуальная модель) — мозг диалога/оркестрации
- Генерация документов: `docxtpl`/Jinja по DOCX-шаблону → PDF (LibreOffice headless или WeasyPrint)
- Крипто: `cryptography` (Fernet) / libsodium
- Геокодинг: **Nominatim** (self-host) + справочник аэропортов/станций
- Внутренняя очередь: на БД (просто) или Redis (опционально)
- **Docker / docker-compose** для деплоя (переносимость VPS)
- **Совместимость:** в начале проекта верифицировать поддержку Python 3.14.x всеми зависимостями
  (aiogram, Playwright, драйвер PostgreSQL, docxtpl) и закрепить версии.

## 10. Фазы развития

- **Фаза 1 (MVP):** роли + онбординг; профиль; понимание запроса для поездок по КЗ; определение точки
  отправления (геолокация/профиль); поиск рейсов+отелей с ранжированием «время → цена в рамках
  политики»; показ вариантов и подтверждение в один тап; бронь браузер-агентом с передачей OTP;
  генерация приказа + доставка в Telegram; контейнер-архив + индекс; история поездок.
- **Фаза 2:** доставка приказа email/WhatsApp; международные направления с уточнением адреса;
  напоминания (регистрация, вылет); новые провайдеры (API/GDS); расширенная политика.
- **Фаза 3:** другие способности секретаря (календарь, встречи, расходы) — каркас уже
  мульти-способностный.

## 11. Открытые вопросы

### Решено
- **Шаблон приказа:** DOCX-документ с размеченными местами для подстановки (docxtpl). Сам файл
  шаблона + реквизиты компании предоставит владелец.
- **Сервисы поиска/покупки:** предпочтительно сервисы банков **Freedom** (Aviata/Freedom Travel) и
  **Kaspi** (Kaspi Travel), где привязана карта.
- **Локальный LLM браузер-агента:** предположительно **OpenClaw** (уточнить требования к GPU на VPS).
- **Геокодер:** **Nominatim** (self-host); источник справочника аэропортов/станций — уточнить
  (например, OurAirports / OpenFlights).
